Reviewed `supabase/migrations/0002_replace_session_details.sql` (`public.replace_session_details(p_session_id uuid, p_sets jsonb, p_distance_m int)`) pre-apply on 2026-07-03, against the live schema from [[momentum_0001_schema_review]] (0001 confirmed applied; its CRITICAL FK-deferrable fix and least-privilege GRANT/REVOKE were confirmed present in the live DB).

**Security verified sound (no CRITICAL found):** SECURITY INVOKER + `search_path=''` + fully schema-qualified identifiers (`public.strength_sets` etc.) means RLS is not bypassed and there is no search_path-hijack / privilege-escalation path. The "delete matches nothing under RLS" reasoning in the migration's own comment is correct — confirmed against the actual 0001 RLS policies (child-table ownership via `EXISTS (... sessions s ... s.user_id = (select auth.uid()))`). A cross-user `p_session_id` either deletes 0 rows (RLS-filtered) or, on insert, fails the `WITH CHECK` policy, aborting the whole call (single transaction, so prior deletes in the same call roll back too — the atomicity goal is genuinely achieved). jsonb parsing uses only typed casts (`::uuid`, `::numeric`, `::integer`), never dynamic SQL, so there is no injection risk; malformed elements fail closed via cast errors or NOT NULL violations. `set_order` via `jsonb_array_elements(...) WITH ORDINALITY` is a sound design — it trivially satisfies the `(session_id, exercise_id, set_order)` unique constraint without depending on client-supplied ordering.

**HIGH finding:** unconditional `DELETE` from both `strength_sets` and `cardio_details` followed by *conditional* re-insert (`if p_distance_m is not null`) is a silent-data-loss footgun. If a client call for a run/swim session omits `p_distance_m` (JS bug, `undefined` key dropped by `JSON.stringify`, etc.), PostgREST/Postgres silently applies the `default null`, the existing `cardio_details` row is deleted, and nothing re-inserts it — no error, no trigger catches it (triggers are insert/update-only, not delete). Fix recommended: look up the session's `sport` first and branch — only touch `strength_sets` when `sport = 'strength'`, only touch `cardio_details` when `sport in ('run','swim')`, and `raise exception` if `p_distance_m is null` in the cardio branch instead of silently skipping the insert.

**MEDIUM findings:** (1) no upfront ownership/existence check on `p_session_id` — a bogus or foreign session_id with an empty payload silently no-ops and returns success, masking client bugs; fix by selecting the session's sport `where id = p_session_id and user_id = (select auth.uid())` and `raise exception` `if not found`. (2) no cap on `p_sets` array length — combine with the sport-lookup refactor to add a `jsonb_array_length(...) > N` guard (e.g. 200), partly to bound the per-row `FOR EACH ROW` sport-consistency trigger cost from 0001 (each inserted row reruns an identical `EXISTS (select ... from sessions ...)` check — cheap per-row via PK/index but O(n) with no batch cap).

**LOW findings:** raw Postgres cast/constraint errors propagate to the API caller (minor internal-detail exposure, not a real vuln); no concurrency guard against two concurrent `replace_session_details` calls for the same session racing (low risk for single-user-edits-own-session usage; unique constraint would catch true corruption, worst case one call errors); `strength_sets.exercise_id` FK validation bypasses RLS on `exercises`, so a user could attach a set to another user's *private* custom exercise if they guessed its UUID — this is pre-existing from 0001 (not introduced by 0002), restated here because 0002 is the first bulk-insert consumer of that column.

**Verified via EXPLAIN (no ANALYZE, non-destructive):** `delete from strength_sets where session_id = ...` uses `strength_sets_session_idx` (bitmap index scan); `delete from cardio_details where session_id = ...` uses the PK index scan. No missing-index issue. Grants (`revoke ... from public, anon; grant execute ... to authenticated`) match 0001's least-privilege pattern. Function is implicitly VOLATILE (correct, does DML) — no explicit volatility keyword needed but worth noting it wasn't declared.

**Why:** So a follow-up review of a 0002 fix-up migration (or the eventual 0003) can check whether the sport-aware branching / upfront ownership check / array-size cap were added, rather than re-deriving this analysis. Also confirms 0001's previously-flagged CRITICAL (FK deferrable) and HIGH (grants) issues were resolved by the time 0002 was authored.

**How to apply:** When reviewing the next migration touching `replace_session_details`, `strength_sets`, `cardio_details`, or `sessions`, check whether the unconditional-delete/conditional-insert footgun was fixed (sport-aware branching), whether an ownership/existence guard was added, and whether a set-count cap exists. If not, keep flagging as HIGH/MEDIUM respectively.
